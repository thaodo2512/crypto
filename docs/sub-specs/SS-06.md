# SS-06: Data Health Monitor

**Master Spec Sections:** §3.6
**Dependencies:** SS-01
**Estimated LOC:** ~120

## Scope

Monitors the health of all data sources: tracks last successful fetch timestamps, response latencies, consecutive failure counts, and overall source status. Provides a health report dict for the Telegram `/health` command and a `check_source()` method that collectors call after each fetch to record success/failure. This sub-spec does NOT send Telegram messages (SS-18), trigger alerts (SS-17), or implement fallback data fetching — it only tracks and reports health status.

## Key Requirements

1. `HealthMonitor` class that tracks status for each data source by name (e.g., `"binance_spot"`, `"deribit_options"`, `"fred_cpi"`, `"alternative_fng"`).
2. `record_success(source: str, latency_seconds: float)` records a successful fetch: updates last success timestamp, resets consecutive failure count to 0, stores latency.
3. `record_failure(source: str, error: str)` records a failed fetch: increments consecutive failure count, stores error message and timestamp.
4. `get_source_status(source: str)` returns a dict with: `last_success` (ISO timestamp or None), `last_failure` (ISO timestamp or None), `consecutive_failures` (int), `latency_seconds` (float or None), `is_stale` (bool), `is_degraded` (bool), `last_error` (str or None).
5. `is_stale` is `True` when `last_success` is more than `staleness_threshold_minutes` ago (from config, default 30 min), or when `last_success` is `None`.
6. `is_degraded` is `True` when `consecutive_failures >= consecutive_failures_before_fallback` (from config, default 3).
7. `get_health_report()` returns a dict of all tracked sources and their statuses. Also includes `overall_healthy` (bool): `True` only if no sources are stale or degraded.
8. `check_latency_warning(source: str)` returns `True` if latest latency exceeds `latency_threshold_seconds` (from config, default 5).
9. All thresholds read from `config/settings.yaml` `health.*`, not hardcoded.
10. In-memory tracking (no DB writes) — health state is ephemeral per process lifecycle. Simple and fast.

## Acceptance Criteria

- [ ] `record_success()` updates last_success timestamp and resets failure count
- [ ] `record_success()` stores latency_seconds
- [ ] `record_failure()` increments consecutive_failures and stores error
- [ ] `get_source_status()` returns correct status dict for known source
- [ ] `get_source_status()` returns default status for unknown source
- [ ] `is_stale` is True when last_success is older than threshold
- [ ] `is_stale` is True when source has never succeeded
- [ ] `is_degraded` is True when consecutive_failures >= threshold
- [ ] `get_health_report()` returns all sources with statuses
- [ ] `get_health_report()` sets overall_healthy=False when any source is stale or degraded
- [ ] `check_latency_warning()` returns True when latency exceeds threshold
- [ ] All thresholds read from config, not hardcoded

## Test Strategy

- **Unit tests** with known timestamps (mock `datetime.now`)
- **Test staleness** by setting last_success to > 30 min ago
- **Test degradation** by calling `record_failure()` 3+ times
- **Test health report** with mix of healthy, stale, and degraded sources
- **Edge cases**: unknown source status, zero latency, exactly-at-threshold values

## Integration Points

**Provides to:**
- SS-17 (Alert System): `get_health_report()` for health alerts
- SS-18 (Telegram Bot): `get_health_report()` for `/health` command response
- SS-21 (Scheduler): `is_degraded()` to decide fallback behavior

**Depends on:**
- SS-01: Nothing from DB (in-memory only), but follows project conventions
- `config/settings.yaml`: `health.staleness_threshold_minutes`, `health.latency_threshold_seconds`, `health.consecutive_failures_before_fallback`
